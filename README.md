Each project lists **what it proves**, the **stack**, and a **stretch goal**. Deploy them, write them up, and use them as referral conversation-starters.

---

## 1. RAG over your own docs 🟢
**Proves:** embeddings, chunking, retrieval, grounding, citations.
**Stack:** Python, an embedding model, a vector DB, an LLM API.
**Stretch:** add re-ranking + a retrieval eval (recall@k).

## 2. LLM eval harness 🟡
**Proves:** you take quality seriously (rare in juniors).
**Stack:** an eval set, LLM-as-judge + human labels, regression tests.
**Stretch:** wire it into CI so prompt changes get scored automatically.

## 3. Tool-using agent 🟡
**Proves:** orchestration, function calling, guardrails, termination.
**Stack:** an agent framework or hand-rolled loop + 2–3 tools.
**Stretch:** add memory + cost/step limits + a trace viewer.

## 4. Structured extraction service 🟢
**Proves:** prompting + validation + reliability.
**Stack:** LLM + schema validation (e.g. Pydantic) + an API.
**Stretch:** add retries, fallbacks, and an accuracy dashboard.

## 5. Production RAG with caching 🔴
**Proves:** system design, latency/cost awareness.
**Stack:** retrieval + semantic cache + monitoring.
**Stretch:** multi-tenant isolation + freshness pipeline.

## 6. Hybrid search 🟡
**Proves:** retrieval depth beyond naive embeddings.
**Stack:** dense + sparse (BM25) + fusion, with a quality comparison.

## 7. Evals-driven prompt optimizer 🟡
**Proves:** rigor; turns "vibes" into measurement.

## 8. Multimodal feature 🟡
**Proves:** breadth — image/audio + text.

## 9. Streaming chat with citations 🟢
**Proves:** real UX + grounding.

## 10. Fine-tune vs RAG comparison 🔴
**Proves:** judgment — you can argue the trade-off with data.

## 11. Guardrails / safety layer 🟡
**Proves:** production maturity (PII, jailbreaks, validation).

## 12. Cost/latency dashboard 🟡
**Proves:** you think like an owner, not just a builder.
